The notification problem, solved

Every product eventually needs notifications. First it's a transactional email. Then a password reset SMS. Then push for mobile. Then in-app alerts. Before long you're maintaining separate integrations for SendGrid, Twilio, Firebase, and your own in-app feed — with no unified way to manage templates, preferences, or delivery history.

Novu solves this at the infrastructure level. One API for all channels, a single dashboard for templates and workflows, and subscriber-level preference management built in. The novu-kotlin SDK wraps all of this in idiomatic Kotlin — type-safe data classes, Retrofit-powered HTTP, and synchronous extension functions for zero-boilerplate integration.

Install

Add the dependency to your Gradle build file:

build.gradle.kts
kotlin // build.gradle.kts
dependencies {
  implementation("io.github.crashiv:novu-kotlin:0.1.6")
}

Initialise the client

kotlin // Minimal setup
val novu = Novu(apiKey = "<YOUR_API_KEY>")

// Custom backend URL (self-hosted Novu)
val novu = Novu(NovuConfig(
  apiKey = "<YOUR_API_KEY>",
  backendUrl = "https://novu.your-company.com/v1/",
))

Trigger your first notification

kotlin val response = novu.trigger(TriggerEventRequest(
  name = "welcome", // template identifier
  to = "user-123", // subscriber ID
  payload = mapOf(
    "firstName" to "Shivam",
    "dashboardUrl" to "https://app.example.com",
  ),
  transactionId = "txn-${UUID.randomUUID()}", // idempotency key
))

println(response.data.acknowledged) // true

Subscribers

A Subscriber is anyone who can receive notifications — a user, a team, a device. Register them once and Novu handles routing to the right channel based on the credentials you attach (email address, phone number, push token, etc.).

Create and update

kotlin // Upsert on every login — keeps credentials fresh
novu.createSubscriber(SubscriberRequest(
  subscriberId = user.id,
  firstName = user.firstName,
  lastName = user.lastName,
  email = user.email,
  phone = user.phone,
  avatar = user.avatarUrl,
))

// Update when profile changes
novu.updateSubscriber(user.id, UpdateSubscriberRequest(
  email = newEmail,
))

Channel credentials

Attach provider-specific credentials to route notifications correctly — for example, a Firebase device token for push notifications.

kotlin novu.updateSubscriberCredentials(
  subscriberId = user.id,
  request = UpdateSubscriberCredentialsRequest(
    providerId = "fcm",
    credentials = ChannelCredentials(
      deviceTokens = listOf(fcmToken),
    ),
  ),
)

Notification feed & unread count

kotlin val feed = novu.subscriberNotificationsFeed(user.id)
val count = novu.subscriberUnseenNotificationsCount(user.id)

// Mark as seen after displaying in-app notifications
novu.markSubscriberFeed(user.id,
  MarkSubscriberFeedAsRequest(seen = true)
)

Topics — notify groups at once

Topics let you group subscribers and fire a single trigger to the entire group. Use them for pricing plans, feature cohorts, or team-level notifications — without building and maintaining your own fan-out logic.

kotlin // Create a topic for pro plan users
novu.createTopic(CreateTopicRequest(
  key = "plan:pro",
  name = "Pro Plan Subscribers",
))

// Add subscribers when they upgrade
novu.addSubscriber("plan:pro",
  AddSubscribersRequest(subscribers = listOf(user.id))
)

// Trigger to the whole topic — Novu fans out
novu.trigger(TriggerEventRequest(
  name = "feature-announcement",
  to = listOf(Topic(topicKey = "plan:pro")),
  payload = mapOf("featureName" to "AI Summaries"),
))
Zero fan-out code

No loops, no goroutines, no queues. One trigger call — Novu delivers to every subscriber in the topic.

Dynamic membership

Add and remove subscribers from topics at runtime. Membership changes are reflected in the next trigger immediately.

Bulk triggers and broadcasts

Beyond single triggers, the SDK provides two patterns for high-volume sends.

Bulk trigger — personalised at scale

Send multiple personalised notifications in a single HTTP round-trip. Each event in the batch can target a different subscriber with a different payload.

kotlin novu.bulkTrigger(BulkTriggerEventRequest(
  events = invoices.map { inv ->
    TriggerEventRequest(
      name = "invoice-due",
      to = inv.subscriberId,
      payload = mapOf(
        "amount" to inv.amount,
        "dueDate" to inv.dueDate,
      ),
    )
  }
))

Broadcast — the whole audience

Broadcast sends a single event to every subscriber in your Novu environment. Good for product announcements, maintenance windows, or platform-wide alerts.

kotlin novu.broadcast(BroadcastEventRequest(
  name = "maintenance-window",
  payload = mapOf(
    "startsAt" to "2026-04-01T02:00:00Z",
    "duration" to "30 minutes",
  ),
))

Subscriber preferences

Notification fatigue is real. Novu lets subscribers control which channels they receive each template on — and the SDK makes it easy to read and write those preferences from your backend.

kotlin // Read what channels the subscriber has enabled
val prefs = novu.subscriberPreferences(user.id)

// Update: disable SMS for this template only
novu.updateSubscriberPreferences(
  subscriberId = user.id,
  templateId = "tmpl_marketing_digest",
  request = UpdateSubscriberPreferencesRequest(
    channels = mapOf(
      "email" to true,
      "sms" to false,
      "push" to true,
    ),
  ),
)

Under the hood

The SDK is structured around 14 Retrofit API interfaces — one per Novu resource domain. Each interface uses Kotlin suspend functions for coroutine-native async calls. On top of those, a set of extension functions wraps each call with runBlocking for synchronous use without any ceremony.

API Responsibilities
EventsApitrigger, bulkTrigger, broadcast, cancel
SubscribersApiCRUD, credentials, preferences, feed, seen/read state
TopicsApicreate, rename, add/remove subscribers, list
NotificationTemplatesApicreate, update, delete, toggle status
NotificationsApilist, get, stats, graph stats
IntegrationsApiconfigure providers (SendGrid, Twilio, FCM…)
MessagesApilist, delete individual messages
EnvironmentsApimanage environments, API keys, widget settings
LayoutsApicreate/manage notification layout templates
ChangesApilist, apply, bulk-apply environment changes
FeedsApicreate and manage in-app notification feeds
ExecutionDetailsApiquery per-notification delivery details
NotificationGroupsApiorganise templates into groups
InboundParseApivalidate MX records for inbound email parsing

HTTP configuration

Authentication is handled transparently via an OkHttp interceptor that injects the ApiKey header on every request. Logging is configurable. The Gson deserializer is set to lenient mode so partial API responses don't throw.

kotlin // RetrofitHelper sets this up automatically
OkHttpClient.Builder()
  .addInterceptor { chain ->
    val req = chain.request().newBuilder()
      .header("ApiKey", config.apiKey)
      .build()
    chain.proceed(req)
  }
  .addInterceptor(HttpLoggingInterceptor())
  .build()
