Subscription billing that stays out of the way

Billow is a provider-agnostic subscription management library for Go — one interface for Stripe, Razorpay, or any gateway. Zero external dependencies.

March 2026 · 10 min read · nulllvoid/billow
// Wire once. Switch providers without touching your business logic.
mgr := billow.New(
  billow.WithProvider(stripeProvider),
  billow.WithStore(pgStore),
  billow.WithWebhookWorkers(4),
)

sub, err := mgr.Subscribe(ctx, billow.SubscribeRequest{
  CustomerID: "cust_123",
  PlanID: "plan_pro_monthly",
  TrialDays: 14,
})

Why does billing logic always leak?

Payment gateway SDKs are sprawling. Every provider has its own types, its own webhook shapes, its own retry semantics. Over time, Stripe-specific structs start appearing in your domain models. Webhook handlers grow into hundred-line functions. Switching providers becomes a project, not a refactor.

Billow gives billing a clean boundary. One Provider interface to implement per gateway. One Manager that handles persistence, caching, metered usage, and async webhook dispatch. Your business logic stays untouched regardless of which payment gateway you run.

Billow requires Go 1.21+ and has zero external dependencies. Bring your own store and provider — everything else is pure Go standard library.

What's in the box

🔌
Provider Interface

One interface to implement for any payment gateway. Swap Stripe for Razorpay — or run both — without touching a single handler.

📦
Pluggable Stores

PlanStore, SubscriptionStore, UsageStore interfaces. Ship with the in-memory implementation, graduate to Postgres.

📊
Metered Usage

Track usage per metric, enforce limits, and report to your provider with the built-in sweeper. No manual fan-out needed.

🪝
Webhook Dispatch

Parse, normalize, and dispatch webhook events via an async worker pool. Register typed handlers — no switch statements required.

⚡
Plan Cache

TTL-based plan caching out of the box. Hot paths don't hit your store for plan lookups.

📈
Metrics Hooks

Prometheus and Datadog-compatible instrumentation. Plug in your existing metrics client and get billing visibility for free.

Get started in 30 seconds

$ go get github.com/nulllvoid/billow

Step 1 — Implement a Provider

The Provider interface is the only gateway-specific code you write. Implement it once for Stripe, once for Razorpay — then never again.

go type StripeProvider struct { client *stripe.Client }

func (p *StripeProvider) CreateSubscription(ctx context.Context, req billow.SubscribeRequest) (*billow.Subscription, error) {
  // call Stripe SDK, map to billow.Subscription
  return &billow.Subscription{...}, nil
}

// ParseWebhook, CancelSubscription, ReportUsage, ...

Step 2 — Choose a Store

Start with the built-in memory store for development. Both the simple and 256-shard variants implement the same interface.

go import "github.com/nulllvoid/billow/store/memory"

store := memory.New() // simple, dev-friendly
// or
store := memory.NewSharded() // 256-shard, high-throughput

Step 3 — Create the Manager

go mgr := billow.New(
  billow.WithProvider(stripeProvider),
  billow.WithStore(store),
  billow.WithPlanCacheTTL(5*time.Minute),
  billow.WithMetrics(prometheusMetrics),
  billow.WithWebhookWorkers(4),
)
defer mgr.Close()

Subscription lifecycle

Billow tracks every subscription through a well-defined state machine. States are stored locally and kept in sync with your provider via webhooks.

State Meaning Transitions to
trialing Free trial period — not yet charged active, canceled
active Paid and in good standing paused, past_due, canceled
paused Billing paused by request active, canceled
past_due Payment failed, grace period active active, canceled, expired
canceled Explicitly canceled —
expired Grace period exhausted —
go // Pause, resume, cancel — all the same pattern
err = mgr.PauseSubscription(ctx, subID)
err = mgr.ResumeSubscription(ctx, subID)
err = mgr.CancelSubscription(ctx, subID, billow.CancelAtPeriodEnd)
err = mgr.UpgradeSubscription(ctx, subID, newPlanID)

Metered billing

Track usage per customer per metric, enforce limits in real time, and let the background sweeper handle provider reporting. No manual cron jobs, no race conditions.

go // Record usage
err = mgr.RecordUsage(ctx, billow.UsageRecord{
  SubscriptionID: subID,
  Metric: "api_calls",
  Quantity: 1,
})

// Check a single limit
ok, err := mgr.CheckLimit(ctx, subID, "api_calls")

// Check multiple limits in one call
results, err := mgr.CheckLimits(ctx, subID, "api_calls", "storage_gb")
Persist-then-Report

Usage is written to your store first, then swept to the provider. Survives provider outages without data loss.

Per-Metric Indexing

Usage is indexed by subscription → metric for fast limit checks, even under high write load.

Webhooks without boilerplate

Register typed event handlers on the Manager. Billow parses the raw webhook payload via your provider, normalizes it to a WebhookEvent, and dispatches it to the right handler — asynchronously, via a bounded worker pool.

go mgr.OnWebhook(billow.EventSubscriptionActivated, func(ctx context.Context, e billow.WebhookEvent) error {
  // provision access, send welcome email, etc.
  return nil
})

// In your HTTP handler:
err = mgr.HandleWebhook(ctx, r.Body, r.Header)

The worker pool size is configurable via WithWebhookWorkers. Graceful shutdown via mgr.Close() drains all in-flight events before exit.
