Build resilient data pipelines in Go — without the boilerplate

DataPipe is a generic, extensible library for chaining multi-source fetchers, enriching data in parallel, and observing every step — all with idiomatic Go generics.

March 2026 · 8 min read · nulllvoid/datapipe
// One pipeline. Three sources. Zero glue code.
chain := datapipe.NewFetcherChain[User, *UserRequest](
  datapipe.ChainWithFetcher(primaryDB),
  datapipe.ChainWithFetcher(replicaDB),
  datapipe.ChainWithFetcher(elasticSearch),
  datapipe.ChainWithFallbackMode(datapipe.FallbackSequential),
)

resp, err := pipeline.Execute(ctx, &UserRequest{Limit: 10})

Why another data layer library?

Almost every backend service in Go eventually accumulates the same accretion of logic: "try the primary DB, fall back to the replica, add user data from a microservice, then maybe hydrate from cache." That logic ends up scattered across handlers, repositories, and service layers — loosely coupled at best, a fragile spaghetti at worst.

DataPipe gives that logic a home. It codifies the fetch-enrich-respond pattern into a composable, type-safe pipeline you can reason about, test independently, and reuse across your codebase. Think of it as the missing abstraction layer between your business logic and your data sources.

DataPipe requires Go 1.21+ and leans fully into generics — no interface{} casting, no runtime surprises. Your pipeline is as typed as your domain model.

Step 1 — Define your domain

DataPipe works with any struct that satisfies the lightweight Entity and Request interfaces.

  ID string
  Name string
}

func (u User) GetID() string { return u.ID }

type UserRequest struct { Limit, Offset int }
func (r *UserRequest) GetLimit() int { return r.Limit }
func (r *UserRequest) GetOffset() int { return r.Offset }

Embed BaseFetcher to get priority, naming, and CanHandle routing for free. Override only what you need.

go type DBFetcher struct {
  datapipe.BaseFetcher[User, *UserRequest]
  db *sql.DB
}

func (f *DBFetcher) Fetch(ctx context.Context, req *UserRequest) ([]User, bool, error) {
  // your SQL query here

Fallback strategies

One of DataPipe's most powerful features is its declarative fallback system. Stop writing manual retry / fallback logic and let the pipeline handle it.

Response
Mode Behavior Best for
FallbackSequential Try each source in priority order, continue if insufficient results DB primary → replica → search
FallbackFirstSuccess Use the first source that returns any data Cache layers, fast short-circuits
FallbackParallel Query all sources concurrently, use highest-priority winner Latency-critical reads

Need to route requests selectively? Implement CanHandle on any fetcher to opt it in or out based on request shape — e.g. send full-text search requests only to Elasticsearch while regular queries go straight to the database.

Enrichment, done right

Enrichment is the step that kills performance in most backends: you fetch a list of entities, then fan out to 4 microservices to hydrate each one. Without structure, this becomes either dangerously sequential or hand-rolled goroutine soup.

DataPipe's parallel enrichment gives you a bounded worker pool, per-enricher auth/expand filtering, and a clean abort path if a required enricher fails.

go fundEnricher := &FundAccountEnricher{
  BaseEnricher: datapipe.NewBaseEnricher[*Payout]("fund_account",
    datapipe.EnricherWithExpandKey("fund_account"), // opt-in via ?expand=
    datapipe.EnricherWithAllowedAuths("private", "admin"), // guard by auth
    datapipe.EnricherWithRequired(false), // soft failure ok
  ),
}
EnricherWithExpandKey

Only enriches when the key is present in the request's expand params. Zero overhead when not needed.

EnricherWithAllowedAuths

Restrict enrichment to specific auth scopes — great for hiding sensitive fields from public callers.

EnricherWithRequired

Mark critical enrichers as required. A failure will halt the pipeline rather than silently returning incomplete data.

Worker Pool Limiting

Pass a concurrency cap when enabling parallel enrichment. No goroutine explosion, no need for external semaphores.

Observability without friction

DataPipe ships with built-in middleware and metrics hooks. Plug in your existing Metrics implementation and every pipeline execution is automatically instrumented — stage latencies, error rates, fallback counts.

go pipeline := datapipe.New(...,
  datapipe.WithMetrics(prometheusMetrics),
  datapipe.WithMiddleware(loggingMiddleware, tracingMiddleware),
)

Middleware wraps the full pipeline execution — useful for distributed tracing spans, request-level logging, and audit trails. Pair it with a structured logger and you get deep visibility into every data fetch your service performs.
